## Offene Issues (vom EN-Volltest am 2026-04-25)

### Issue #1 — Quick-Win: Footer-Links auf Landing ohne BASE_URL
**Severity**: Medium · **Aufwand**: 5 Min

`src/pages/LandingPage.tsx` Zeilen 48, 51, 54, 57:
```tsx
<a href="/app/privacy" ...>
<a href="/app/privacy" ...>     // 2× (Privacy + Legal)
<a href="/app/faq" ...>
<a href="/app/sponsors" ...>
```

**Fix-Option A** (minimal, konsistent mit heutigem Refactor):
```tsx
<a href={`${import.meta.env.BASE_URL}app/privacy`} ...>
```

**Fix-Option B** (besser, idiomatisch React-Router):
```tsx
import { Link } from 'react-router-dom'
<Link to="/app/privacy" ...>
```
Option B ist eleganter weil React-Router automatisch den basename voranstellt — aber Option A ist näher am bestehenden Code-Pattern (heutige andere Refactors nutzen `import.meta.env.BASE_URL`).

### Issue #3 — Cookie-Banner-Locale-Sync-Bug
**Severity**: Medium · **Aufwand**: ~30 Min

**Reproduktion**: Auf Landing initial DE → Sprachschalter auf EN klicken → Cookie-Banner bleibt DE bis zum Page-Reload.

**Root-Cause**: `src/hooks/useLocale.ts` ist ein normaler `useState`-Hook, kein singleton/Context. `App.tsx:50` und `LandingPage.tsx:12` rufen jeweils `useLocale()` separat auf und haben getrennte State-Instances. `setLocale()` in LandingPage updated localStorage + den lokalen State, aber App.tsx (das `<CookieConsent locale={locale}>` rendert) bekommt das nicht mit.

**Fix-Optionen**:
- **A** — `useLocale` in einen React Context-Provider migrieren, in `main.tsx` um die App wrappen
- **B** — `storage`-Event-Listener im useLocale-Hook für Cross-Instance-Sync
- **C** — globales State-Tool (Zustand, Jotai) für locale

Option A ist am idiomatischsten für React 19. Bringt Cleanup beim Refactor mit (alle Komponenten die `useLocale()` aufrufen, behalten dasselbe API — nur Context-Provider drumherum).

### Issue #4 — Sponsor-Daten halb-DE im EN-Modus
**Severity**: Low · **Aufwand**: variabel

`/app/sponsors` zeigt Sponsor-Beschreibungen wie „Halbtags-Workshop AI-Integration fuer dein Team" auch in EN-Modus. Außerdem ASCII-Umlaut „fuer" statt „für" — Encoding-Issue beim Datenbank-Eintrag.

**Vermutung**: Sponsor-Daten in Supabase ohne i18n-Schema (eine `description`-Spalte statt `description_de`/`description_en`). Möglicher Fix:
- Schema-Migration: `description` → `description_de` + `description_en`
- Edge-Function `get-sponsors` liefert basierend auf User-Locale
- Frontend zeigt entsprechend

ODER: Pragmatisch nur die Umlaute korrigieren und Workshop-Titel zweisprachig stehen lassen.

**Bonus-Issue**: ASCII-Umlaute in DB. Würde mit der i18n-Migration mit-gefixt werden.

### Hinweis (kein Bug) — „AI Dirigent" für Level 6
`src/lib/constants.ts:80` setzt für `de` und `en` denselben Wert „AI Dirigent". Bewusste Branding-Entscheidung (deutsches Wort als Marken-Element). Wenn das ein Bug sein soll: Standard wäre „AI Conductor" oder „AI Maestro" für EN. Andernfalls so lassen.
